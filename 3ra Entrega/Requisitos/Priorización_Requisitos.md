# REQUISITOS FUNCIONALES  

## Método de priorización MoSCoW

| **ID** | **Requisito** | **Descripción** | **Tipo de priorización** | **Justificación** |
|--------|----------------|-----------------|---------------------------|--------------------|
| **RF3** | **Comparación de Precios en Tiempo Real** | El sistema debe mostrar una tabla comparativa por producto que incluya el precio por tienda, unidad de medida (kg/litro), porcentaje de diferencia, disponibilidad y distancia a la tienda. | **Must Have** | Es el núcleo de la aplicación, debido a que representa la funcionalidad principal que aporta valor para el usuario. Permite comparar precios de un mismo producto entre diferentes supermercados en tiempo real, ayudando a identificar dónde conviene comprar. Sin esta función, la aplicación perdería su valor como herramienta de ahorro y eficiencia para el consumidor. |
| **RF7** | **Asistente Virtual Inteligente** | El sistema debe incluir un chatbot capaz de interpretar lenguaje natural, responder preguntas sobre precios y productos, ofrecer recomendaciones y mantener el contexto de la conversación. | **Must Have** | Transforma la aplicación y la hace evolucionar de una simple herramienta de comparación de precios a un asistente personal de compras inteligente. Este chatbot permite al usuario interactuar mediante lenguaje natural, realizar búsquedas personalizadas y recibir recomendaciones adaptadas a sus necesidades, así como que es la característica que diferencia a la aplicación del resto del mercado y potencia su valor innovador. |
| **RF2** | **Historial de Precios** | El sistema debe almacenar los cambios de precio de cada producto durante un periodo de 6 meses y mostrar gráficas que representen la evolución histórica de dichos precios. | **Must Have** | Es el motor de la inteligencia de la aplicación ya que permite analizar la evolución de los precios a lo largo del tiempo, además, gracias a esta información, el sistema puede generar patrones, ofrecer recomendaciones más precisas y anticiparse a posibles variaciones de costo. Esto es lo que posibilita que la aplicación evolucione de una herramienta básica a un asesor de compras verdaderamente inteligente y proactivo. |
| **RF9** | **Gestión de Usuarios y Preferencias** | El sistema debe permitir a los usuarios registrarse e iniciar sesión, así como configurar sus preferencias: supermercados favoritos, ubicación habitual, productos seguidos y alertas personalizadas. | **Must Have** | Es un componente esencial para ofrecer una experiencia personalizada dentro de la aplicación. Permite que cada usuario cree su perfil, guarde productos, configure alertas y reciba recomendaciones adaptadas a sus intereses y hábitos de compra. Sin esta funcionalidad, la aplicación no podría brindar un servicio individualizado ni retener usuarios, por lo que se considera un elemento clave para el correcto funcionamiento del sistema y la fidelización del consumidor. |
| **RF6** | **Geolocalización y Recomendación Inteligente** | El sistema debe solicitar la ubicación del usuario (previa autorización) y calcular la distancia a cada tienda. Debe ofrecer tres modos de recomendación: “Mejor Precio”, “Más Cercano” y “Balance”. | **Must Have** | Es una funcionalidad esencial para cumplir con el propósito central de la aplicación: ayudar al usuario a encontrar dónde le conviene comprar. No solo compara precios, sino que integra la ubicación geográfica como un factor clave en la decisión de compra, permitiendo identificar las tiendas más cercanas y económicas. Esta capacidad convierte la aplicación en una herramienta realmente útil y práctica para el consumidor, por lo que se considera indispensable dentro del sistema. |
| **RF10** | **Sistema de Notificaciones** | El sistema debe enviar notificaciones (web, móvil o email) cuando un producto seguido baje de precio, tenga una oferta especial o alcance el precio objetivo definido por el usuario. | **Must Have** | El sistema de notificaciones es fundamental para alertar al usuario sobre cambios relevantes en los precios, nuevas ofertas o productos de interés. Esta funcionalidad garantiza que el usuario reciba información en tiempo real, aumentando la eficacia de la aplicación y fomentando su uso continuo. Sin un sistema de notificaciones la comparación de precios y las oportunidades de ahorro no podrían ser aprovechadas de manera oportuna. |
| **RF8** | **Validación con Datos de Profeco** | El sistema debe consultar semanalmente la base pública “Quién es Quién en los Precios” de Profeco y mostrar un sello de validación en los productos que coincidan con dicha fuente. | **Should Have** | Este requisito añade un nivel de credibilidad y validación importante a la aplicación. Aunque la app puede operar sin esta información, contar con la verificación de precios por parte de un organismo oficial constituye un diferenciador significativo frente a la competencia, aumentando la confianza del usuario en la información presentada. |
| **RF11** | **Gestión de Listas de Compra** | El sistema debe permitir crear, editar y eliminar listas de compra, mostrando el costo total por supermercado, el ahorro potencial y los productos no disponibles. | **Should Have** | Permite al usuario planificar y controlar sus compras, registrando productos y llevando un seguimiento de estos. Aunque no es esencial para la comparación de precios, mejora significativamente la experiencia de usuario y la utilidad general de la aplicación. |
| **RF5** | **Validación de Confiabilidad de Ofertas** | El sistema debe analizar el historial de ofertas para clasificar su confiabilidad en tres niveles (ALTA, MEDIA, BAJA) según la duración esperada y mostrar el descuento real aplicado. | **Could Have** | Esta funcionalidad utiliza datos históricos para evaluar la confiabilidad de las ofertas disponibles, aportando un nivel adicional de inteligencia al sistema. Aunque mejora la precisión y la confianza en la información presentada al usuario, no es un requisito esencial para la operación básica de la aplicación. |
| **RF12** | **Reporte de Discrepancias** | El sistema debe incluir un botón para reportar precios incorrectos, permitiendo al usuario enviar la tienda, el precio real, evidencia (foto) y comentarios. | **Could Have** | Es fundamental para garantizar la transparencia y confiabilidad de la información que ofrece la aplicación. Permite a los usuarios reportar errores, precios incorrectos o datos desactualizados en tiempo real, fortaleciendo la precisión del sistema. Además, contribuye al control de calidad de la plataforma, asegurando que los datos mostrados sean siempre verídicos y actualizados, lo cual es esencial para mantener la credibilidad y confianza del usuario. |
| **RF1** | **Integración de APIs de Supermercados** | El sistema debe conectarse automáticamente a las APIs de cinco supermercados (Walmart, Soriana, Chedraui, Bodega Aurrera y La Comer) para obtener precios actualizados, disponibilidad de productos y ubicación de tiendas. | **Won’t Have** | Este requisito implica una integración técnica avanzada y la dependencia de fuentes externas, lo que requiere acuerdos con supermercados, manejo de grandes volúmenes de datos y mantenimiento constante. Debido a la complejidad técnica y los recursos necesarios que son indispensables para su funcionamiento, no es verificable en el sistema. |
| **RF4** | **Predicción de Precios con IA** | El sistema debe utilizar inteligencia artificial para predecir la variación de precios en los próximos 7, 14 y 30 días, mostrando gráficas con el nivel de confianza del modelo. | **Won’t Have** | Este requisito corresponde a actividades más avanzadas en el sistema debido a que requiere el uso de modelos predictivos y una base de datos histórica sólida. Aunque aportaría gran valor analítico, su desarrollo exige tiempo y procesamiento de datos que exceden el alcance actual del proyecto, por lo que no es verificable. |

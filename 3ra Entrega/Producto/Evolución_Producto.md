#  Evolución del producto

| **ÁREA** | **PRIMERA ENTREGA** | **SEGUNDA ENTREGA** | **TERCERA ENTREGA** |
|---------|----------------------|----------------------|----------------------|
| **DISEÑO** | - El sistema SmartPrice contaba con una definición general de objetivos y estructura.<br>- No existía representación visual del sistema.<br>- No había una descripción clara del funcionamiento interno por módulos.<br>- Los requisitos y las historias de usuario eran generales y sin criterios de aceptación.<br>- No se contaba con un prototipo que mostrara el flujo del usuario. | - Se redefinieron y precisaron los requisitos funcionales y no funcionales, incluyendo criterios de aceptación.<br>- Se actualizaron historias de usuario con nuevos escenarios derivados del análisis del equipo.<br>- Se repriorizaron requisitos para alinearlos con las necesidades reales del usuario.<br>- Se actualizaron casos de uso y el diagrama principal del sistema con mayor coherencia.<br>- Se creó un mockup de media fidelidad con las siguientes secciones: pantalla de inicio, búsqueda/comparación de precios, mapa de tiendas, carrito de compras, perfil de usuario y asistente virtual.<br>- Se validó el prototipo con 16 usuarios (22–35 años).<br>- Se realizaron ajustes visuales basados en su feedback (botones, contraste, distribución). | - Se revisaron y mejoraron los requerimientos no funcionales para corregir tiempos y aspectos subjetivos.<br>- Se refinó el modelo completo de casos de uso con diagramas y descripciones detalladas.<br>- Se ajustaron historias de usuario eliminando ambigüedades y elementos subjetivos.<br>- Se actualizó la priorización de requisitos para clasificar mejor la dificultad e identificar requerimientos complejos (como el de Profeco o APIs).<br>- Se refinaron los perfiles de usuario para representar y describir mejor al público objetivo.<br>- Se mejoró integralmente el prototipo con aprendizajes previos.<br>- Se verificó preliminarmente el cumplimiento de requisitos en las pantallas del prototipo.<br>- Se definió el protocolo de pruebas junto con criterios e indicadores basados en los requisitos de usabilidad.<br>- Se aplicaron pruebas a 21 usuarios (18–49 años) mediante un protocolo estructurado.<br>- Se registraron y analizaron observaciones y datos obtenidos durante las pruebas.<br>- Se redactaron propuestas de mejora aportadas por usuarios y el equipo.<br>- Se elaboró documentación reflexiva sobre competencias específicas y genéricas, además de una crítica constructiva del proceso. |
| **ESTRUCTURA DEL PROYECTO** | - No se utilizaba una metodología formal.<br>- Las actividades eran generales y con organización básica.<br>- No existía un sistema de seguimiento.<br>- La documentación era simple (Word/PDF).<br>- No había planificación clara. | - Se adoptó Scrum y se elaboró un product backlog priorizado y visible.<br>- El trabajo se dividió en cuatro sprints semanales con tareas, responsables y entregables.<br>- Se creó una bitácora digital al final de cada sprint con avances, acuerdos y pendientes.<br>- Se integró GitHub Projects para monitorear progreso, detectar bloqueos y coordinar actividades. | - Se consolidó Scrum con 4 sprints utilizando la aplicación Monda, definiendo actividades, fechas, roles y responsables de revisión.<br>- Se mejoró la documentación técnica del producto mediante un README mejor estructurado y una mejor organización del repositorio.<br>- Se incorporó una métrica de contribución individual basada en las tareas asignadas.<br>- Se consolidó una bitácora final del proceso completo y los aprendizajes del equipo. |
